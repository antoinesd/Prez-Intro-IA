== 1950 — La question de Turing

[.column]
--
[discrete]
=== "Can machines think ?"

Alan Turing pose la question en 1950 dans _Computing Machinery and Intelligence_.

Il propose le **Test de Turing** : si un humain ne peut pas distinguer la machine de l'humain dans une conversation, la machine est "intelligente".

*Ce test est-il la bonne mesure ?*

→ Non, mais il a structuré 70 ans de recherche.

== Partie 2 +
Comment fonctionnent les Transformers

== Intuition : prédire le prochain mot

[.prediction-demo]
--
*Exercice mental* :

> Pour faire une omelette, il faut d'abord casser des **___**

Vous avez deviné : *œufs* — et probablement avec un haut degré de confiance.

*C'est exactement ce que fait un LLM* — mais sur des milliards d'exemples de texte.

[.key-insight]
Un modèle entraîné uniquement à prédire le prochain token apprend implicitement la syntaxe, la sémantique, les faits du monde... et même le raisonnement.

*Pourquoi ?* Parce que pour prédire correctement, il faut comprendre.
--

== Les embeddings : les mots dans l'espace vectoriel

[.key-insight]
En pratique :

* Chaque token devient un vecteur dense (appris)
* La proximité géométrique approxime la proximité de sens
* Les relations sémantiques émergent des données

=== Embeddings : représentation vectorielle

--
Chaque mot est projeté dans un espace de grande dimension (N = 768, 1024, 4096...).

[stem]
++++
\begin{aligned}
\text{chat}    &\mapsto [0.23, -0.87, 0.41, \ldots] \\
\text{chien}   &\mapsto [0.25, -0.84, 0.39, \ldots] && (\text{proche}) \\
\text{voiture} &\mapsto [-0.71, 0.12, -0.55, \ldots] && (\text{loin})
\end{aligned}
++++

Ces vecteurs sont *appris* automatiquement pendant l'entraînement.
--

=== Embeddings : relations et visualisation

*Propriété remarquable* : certaines relations s'expriment comme des déplacements vectoriels.

[stem]
++++
\begin{aligned}
\text{roi} - \text{homme} + \text{femme} &\approx \text{reine} \\
\text{Paris} - \text{France} + \text{Italie} &\approx \text{Rome}
\end{aligned}
++++

image::embeddings_2d.svg[Embeddings en 2D, 700]
[.caption]
Projection 2D d'embeddings (t-SNE) : les mots sémantiquement proches sont proches dans l'espace vectoriel

[NOTE.speaker]
--
Les embeddings sont la première couche du transformer. Ils transforment chaque token (mot ou sous-mot) en vecteur dense. Ces vecteurs encodent le sens de façon distribuée sur toutes les dimensions — pas une dimension = une feature.
--

== L'attention : "qui regarde qui" ?

[.key-insight]
L'attention répond à une question simple :

* Pour chaque token, quels autres tokens sont pertinents ici ?

=== Attention : intuition

Avant les Transformers, les RNN traitaient le texte séquentiellement.
Avec la self-attention, tous les tokens sont traités en parallèle.

[.example-box]
====
> "Le **chat** s'est endormi dans le fauteuil parce qu'**il** était fatigué"

* Pour résoudre "il" → forte attention sur "chat"
* Pour comprendre "fatigué" → attention sur "chat" et "endormi"
====

=== Attention : calcul et interprétation

*Formule de l'attention* :

[stem]
++++
\operatorname{Attention}(Q, K, V) = \operatorname{softmax}\left(\frac{QK^\top}{\sqrt{d_k}}\right)V
++++

Q = Query, K = Key, V = Value

image::attention_viz.svg[Attention, 700]
[.caption]
Carte d'attention : chaque cellule montre le poids d'attention entre deux tokens

[NOTE.speaker]
--
La division par √d_k évite que les produits scalaires ne deviennent trop grands avec des vecteurs de grande dimension, ce qui ferait disparaître les gradients (saturation du softmax).
--

== L'attention multi-têtes : regarder sous différents angles

[.key-insight]
Une seule tête = une seule perspective.
Le multi-head combine plusieurs perspectives en parallèle.

=== Multi-Head : principe

Au lieu d'une seule attention, le Transformer utilise H têtes en parallèle.

Chaque tête apprend une projection différente de la séquence.

Les sorties de toutes les têtes sont *concaténées* puis reprojetées.

=== Multi-Head : ce que les têtes capturent

Exemples typiques de spécialisations :

* syntaxe (sujet-verbe)
* co-références (il/le chat)
* dépendances longues distance
* relations temporelles (avant/après)

=== Multi-Head : ordre de grandeur

[.key-insight]
stem:[\text{GPT-4} : 96\ \text{têtes/couche} \times 96\ \text{couches} \approx 9\,216\ \text{regards simultanés}]

image::multihead_attention.svg[Multi-head attention, 700]
[.caption]
Multi-Head Attention : projections parallèles puis concaténation

== Génération de texte : chercher le meilleur token

[.key-insight]
Trois idées pour lire cette partie :

* Le décodage choisit une séquence, pas seulement un mot
* La température et le sampling contrôlent le style de sortie
* Le pré-entraînement + alignment expliquent le comportement final

=== Décodage : greedy vs beam search

*Greedy search* : choisir à chaque étape le token le plus probable.

[.warning-box]
Limite : un choix local optimal peut produire une phrase globale moins bonne.

*Beam search* : explorer N séquences en parallèle, puis garder la meilleure séquence globale.

image::beam_search.svg[Beam search, 1000]

=== Sampling : température, top-k, top-p

*Température* (créativité) :

* T → 0 : déterministe
* T = 1 : distribution naturelle
* T > 1 : plus créatif, plus risqué

*Top-k sampling* : tirage aléatoire parmi les k tokens les plus probables.

*Top-p (nucleus) sampling* : tirage dans le plus petit ensemble couvrant p% de probabilité cumulée.

=== Entraînement : 3 phases + scaling laws

*Phase 1 — Pre-training* : prédire le prochain token sur un très grand corpus.

*Phase 2 — SFT* : apprendre à suivre des consignes.

*Phase 3 — RLHF* : aligner les réponses sur les préférences humaines.

[.key-insight]
stem:[\text{Performance} \approx f\!\left(N_{\text{params}} \times N_{\text{tokens}} \times \text{compute}\right)]

image::scaling_laws.svg[Scaling laws, 600]
[.caption]
La performance améliore de façon prévisible (loi puissance) avec le compute

[NOTE.speaker]
--
La température est l'un des paramètres les plus importants pour contrôler le comportement d'un LLM en production. T faible pour les sorties structurées, T plus élevée pour la génération créative.

Le pre-training reste la phase la plus coûteuse ; SFT + RLHF permettent ensuite une adaptation domaine plus abordable.
--

== Limites et hallucinations

[.key-insight]
Les LLMs peuvent être impressionnants *et* se tromper avec assurance.

[discrete]
=== Pourquoi les hallucinations existent ?

Un LLM n'est pas un moteur de recherche : il prédit la suite la plus probable.

Conséquence : il peut produire des faits inventés, sources fictives ou chiffres erronés.

=== Réduction du risque (sans garantie totale)

* grounding par sources vérifiées (RAG)
* alignment via feedback humain (RLHF)
* contrôles applicatifs (validation, citations, garde-fous)

[.warning-box]
Règle opérationnelle : vérifier systématiquement faits, dates, chiffres et références.

=== Capacités émergentes

À grande échelle apparaissent aussi :

* raisonnement en chaîne
* analogies inter-domaines
* génération de code
* traduction sans supervision explicite

[.key-insight]
Ces capacités n'ont pas été codées explicitement : elles émergent de l'entraînement prédictif.

== RAG : donner une "mémoire externe" au LLM

[.key-insight]
RAG = *Retrieval-Augmented Generation* : on fait d'abord une recherche dans une base de connaissances, puis on demande au LLM de répondre avec ce contexte.

Version vulgarisée :

* Un LLM seul répond de mémoire (parfois bien, parfois faux)
* Un système RAG lui "ouvre le classeur" avant qu'il réponde
* La réponse est plus fiable et plus traçable

=== RAG : fonctionnement en 4 étapes

[.columns]
--
[.column]
====
[discrete]
=== 1) Préparer la base

On découpe les documents en petits morceaux (_chunks_), puis on les convertit en vecteurs (_embeddings_).

[discrete]
=== 2) Chercher les passages utiles

La question utilisateur est vectorisée, puis on récupère les *k* passages les plus proches dans la base vectorielle.
====
[.column]
====
[discrete]
=== 3) Construire le prompt

On injecte question + passages retrouvés dans le prompt envoyé au LLM.

[discrete]
=== 4) Générer la réponse

Le LLM répond en s'appuyant sur le contexte fourni (et idéalement avec citations de sources).
====
--

=== Architecture type d'un RAG

image::rag_pipeline.svg[Pipeline RAG]

[.caption]
Pipeline simplifié : indexation des documents, retrieval, prompt augmenté, génération de réponse.

=== RAG : ce que ça améliore... et ses limites

[.key-insight]
Le RAG *réduit* les hallucinations, mais ne les supprime pas totalement.

Ce que ça améliore :

* Réponses plus factuelles sur un domaine précis
* Possibilité de citer les sources utilisées
* Mise à jour des connaissances sans réentraîner le modèle

Limites à garder en tête :

* Mauvais retrieval = mauvaise réponse
* Contexte trop long ou bruité = confusion
* Toujours besoin de validation humaine pour les cas critiques

[.warning-box]
Règle pratique : qualité d'un RAG = qualité des documents + qualité de l'indexation + qualité du retrieval.

== AlphaFold : résoudre un problème vieux de 50 ans

[.columns]
--
[.column]
====
[discrete]
=== Le problème du repliement des protéines

*La séquence d'acides aminés* détermine la structure 3D de la protéine.
Mais prédire cette structure depuis la séquence = *problème NP-difficile* (espace de conformations astronomique).

*Levinthal's paradox (1969)* :
Si une protéine de 100 aa testait aléatoirement chaque conformation à 10^13 /s → 10^87 ans pour se replier.
Elle le fait en *microsecondes*. Pourquoi ? L'évolution a trouvé un chemin guidé.

*50 ans de cristallographie aux rayons X et cryo-EM* → ~180 000 structures dans la PDB

*AlphaFold2 (2021)* → prédit avec précision atomique pour ~200 millions de protéines

====

====

== AlphaFold : comment ça marche

[.columns]
--
[.column]
====
[discrete]
=== Architecture hybride

*Entrée* : séquence d'acides aminés + alignements multiples (MSA)

*Evoformer* (cœur d'AlphaFold2) :

* Transformer modifié qui traite simultanément :
** La séquence (1D)
** La matrice de distances entre résidus (2D)
* Itérations de "communication" entre les deux représentations

*Structure module* :

* Converge vers des coordonnées 3D
* Utilise une représentation en "cadres rigides" (SE(3))

*Résultat* : coordonnées atomiques + score de confiance (pLDDT)

====
[.column]
====
[discrete]
=== Impact réel

[.impact-list]
* *Malaria* : nouvelles cibles thérapeutiques identifiées

== Améliorer le raisonnement : au-delà de la prédiction de token

[.columns]
--
[.column]
====
[discrete]
=== Chain-of-Thought : penser étape par étape

*Problème* : les LLMs ont du mal avec le raisonnement multi-étapes

*Solution* : laisser le modèle "penser à voix haute"

[.example-box]
*Sans CoT* :
Q: 17 × 23 = ?
R: 371 ✗

*Avec CoT* :
Q: 17 × 23 = ? Réfléchissons étape par étape.
R: 17 × 20 = 340. 17 × 3 = 51. 340 + 51 = 391 ✓

*OpenAI o1 / o3 (2024)* : entraîné spécifiquement à raisonner
→ Performances proches des experts en maths olympiques et coding

====
[.column]
====
[discrete]
=== Reasoning as Test-Time Compute

[.key-insight]
Idée centrale : au lieu de dépenser tout le compute à l'entraînement, allouer plus de compute à l'inférence (réfléchir plus longtemps)

*Self-Consistency* : générer N raisonnements, voter pour la réponse majoritaire

*Tree of Thoughts* : explorer un arbre de raisonnements possibles

*Process Reward Model* : évaluer la qualité de chaque étape intermédiaire, pas seulement la réponse finale

→ La "pensée lente" de Kahneman appliquée aux LLMs

== Défis et limites actuelles

[.columns.challenges]
--
[.column]
====
[discrete]
=== Limites techniques

*Hallucinations* : génèrent des informations fausses avec confiance → risque critique en science

*Contexte limité* : même avec 1M tokens (Gemini), les modèles "oublient" en dehors du contexte

*Coût énergétique* : GPT-4 query ~0.001-0.01 kWh → ~10× une recherche Google. À l'échelle mondiale...

*Biais* : héritent des biais des données d'entraînement — corpus scientifique anglophone dominant

====
[.column]
====
[discrete]
=== Défis éthiques et réglementaires

*Droits d'auteur* : les modèles entraînés sur des articles scientifiques — légal ?

*Reproductibilité* : les modèles propriétaires ne sont pas auditables

*Impact emploi* : automation de tâches cognitives qualifiées

*IA Act (EU 2024)* : premier cadre réglementaire mondial pour l'IA à haut risque
